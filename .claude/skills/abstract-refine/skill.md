# Abstract Refine

포지셔닝 결정을 초록에 녹여 재작성하고, self-review로 품질을 보증하는 스킬.

## 언제 사용하는가

- `research-positioning` 으로 방향이 정해진 직후
- 기존 초록의 주장-근거 대응이 흐트러졌을 때
- 리뷰어에게 걸릴 과잉 주장을 사전 제거하고 싶을 때

## 전제

- `_workspace/positioning_decision.md` (필수)
- `_workspace/gap_matrix.md` (과잉 주장 판정용)
- `_workspace/prior_work_table.md` (중복 문구 감지용)
- `초록_KITS.md` (원본, **읽기 전용**)

## 불변 원칙

1. **원본 파일 보호**: `초록_KITS.md` 는 절대 수정하지 않는다. 결과는 `_workspace/abstract_v2.md` 에 쓴다. 최종 승인·병합은 사용자 몫.
2. **주장-근거 대응**: 초록의 모든 주장은 `positioning_decision.md` 의 선택된 옵션 또는 실험 스케치에 근거가 있어야 한다.
3. **HR 스키마-온리 제약 명시**: 실험이 실데이터가 아닌 스키마 이식이라는 사실을 반드시 한 문장 이상으로 명시. 숨기면 리뷰 과정에서 역효과.

## 재작성 가이드 — 6 구성요소 흐름

초록 본문은 한 문단으로 다음 흐름을 따른다:

1. **문제 맥락** (1~2문장): Text-to-SQL 자기교정이 왜 중요한가
2. **선행연구의 진짜 빈틈** (1~2문장): gap_matrix의 `진짜 gap`/`부분 gap` 만. 일반화 금지
3. **제안의 핵심 thesis** (1~2문장): 선택된 포지셔닝
4. **방법의 차별점** (2~3문장): 어떻게 gap을 메우는지 구체적으로
5. **실험 설계** (2~3문장): HR 스키마-온리 이식 방법 + 벤치마크 + 베이스라인 + 어블레이션
6. **기대 기여** (1문장): gap 강도에 비례하는 톤으로

목표 길이: 450~600자 (한국 학회 초록 관행).

## 과잉 주장 단어 규칙

다음 단어는 gap_matrix에서 **진짜 gap** 으로 분류된 기여에만 사용:
- "최초", "혁신", "완전히 새로운", "패러다임 전환", "근본적"

`부분 gap` 에 대응하는 기여는 다음 톤으로:
- "기존 접근을 확장", "보완", "정교화", "통합", "체계화"

`이미 존재` 로 분류된 항목은 **기여로 주장하지 말 것**.

## 중복 문구 감지

`prior_work_table.md` 의 각 논문 method 필드와 재작성한 초록 문장의 어휘 유사성을 점검:
- 동일 동사구 재사용 시 (예: "leverages LLM for self-correction with feedback") 재표현
- 선행연구 고유 표현을 그대로 가져다 쓰면 안 됨

## Self-Review 체크리스트

재작성 직후 즉시 실행하고, 문제 발견 시 수정 후 재검사:

- [ ] 초록의 각 주장이 `positioning_decision.md` 에 근거 있는가?
- [ ] `prior_work_table.md` 의 method 설명과 문구가 거의 동일한 곳이 없는가?
- [ ] `이미 존재` 로 판정된 기여를 여전히 차별점으로 내세우고 있지 않은가?
- [ ] HR 스키마-온리 실험 설계가 한 문장 이상으로 명시되었는가?
- [ ] 과잉 주장 단어가 진짜 gap에만 쓰였는가?
- [ ] 한 문단, 450~600자 범위인가?
- [ ] 주제어가 재작성된 포지셔닝과 일치하는가?

```markdown
---

- {주요 변경점 2}
- ...

- **약화**: {어떤 표현을 왜 약화했는가}
- **신규**: {어떤 내용을 왜 추가했는가}

## Self-Review 결과

- [x] 체크리스트 항목 2
- ...

```

## 에러 핸들링

- `positioning_decision.md` 에 `NEEDS_EXPERIMENT_REDESIGN` 또는 `NEEDS_NEW_ANGLE` 플래그가 있으면: 재작성을 시도하지 말고 `abstract_v2.md` 에 "포지셔닝 재검토 필요 — 초록 재작성 보류" 와 플래그 이유만 남기고 종료.
- Self-review에서 같은 문제가 2회 이상 수정해도 해결 안 되면: 해당 항목을 `UNRESOLVED` 로 표시하고 사람의 개입이 필요함을 출력.
