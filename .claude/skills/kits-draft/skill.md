# KITS Draft — 한국어 KITS 논문 집필 가이드

자기교정 Text-to-SQL 논문의 KITS 양식 한국어 본문 작성 규칙.

## 핵심 원칙

**초록은 헌법이다.** [docs/초록.md](../../../docs/초록.md)의 주장·RQ·용어를 본문이 반박·확장하는 것은 괜찮지만 **모순**은 허용되지 않는다. 초록과 모순되면 본문을 수정하고 초록은 그대로 둔다.

**한국어 학술 톤**. 영어 직역투·과잉 주장 배제. "~임이 확인되었다", "~로 해석될 수 있다" 같은 관찰 기반 표현 선호. "최초", "완벽", "모든 상황에서"는 금지.

## 섹션 구조 (표준 6장 + 참고문헌)

| 장 | 제목 | 분량 목표 | 핵심 내용 |
|----|------|----------|----------|
| 1 | 서론 | 약 1쪽 | 배경·동기, RQ1/RQ2, 기여 3~4개, 논문 구성 |
| 2 | 관련 연구 | 약 1.5쪽 | 4축(오류분류/의미검증/유형별 프롬프트/반복 교정) 구조로 선행연구 정리 |
| 3 | 제안 방법 | 약 2쪽 | 파이프라인 5모듈, NLI 의도 일치 검증(RQ1), 유형별 교정(RQ2), 오류 분류 체계 |
| 4 | 실험 | 약 2쪽 | 데이터셋(BIRD+HRDB), 베이스라인, 지표, 주 결과, 어블레이션 |
| 5 | 토의 | 약 0.5~1쪽 | RQ 재확인, 한계, 일반화 범위, 위협 타당성 |
| 6 | 결론 | 약 0.3쪽 | 기여 요약, 향후 과제 |
| - | 참고문헌 | - | KITS 스타일 |

정확한 규정은 [_workspace/05_KITS_submission_guidelines.md](../../../_workspace/05_KITS_submission_guidelines.md) 우선. 학회 사이트 원문과 불일치 시 원문 우선.

## 섹션별 집필 체크리스트

### 1. 서론
- [ ] 기업 데이터 접근성 문제로 시작 (초록과 일치)
- [ ] 기존 자기교정이 코드 실행 측면에 치우친 문제 지적
- [ ] RQ1/RQ2 명확히 서술
- [ ] 기여 3~4개 bullet
- [ ] "본 논문은 2장에서 관련 연구를, 3장에서..." 논문 구성 문단으로 마무리

### 2. 관련 연구
4축으로 서브섹션 구성:
- 2.1 오류 분류 체계 (MapleRepair, SQL-of-Thought, SQLens)
- 2.2 SQL 의미·실행 검증 (CHESS, SQLDriller, SQLens)
- 2.3 유형별 교정 지시문 (Fan-in: 대부분의 선행연구가 **단일 공용 지시문** 사용)
- 2.4 반복 교정 루프 (DIN-SQL, MAC-SQL Refiner)
- 2.5 본 연구의 위치: **4축 모두 결합 + NL 공간 의미검증 신규 도입**

인용은 prior_work_table·lit_*.md 기반만. 새 논문은 연구자에게 요청 후 추가.

### 3. 제안 방법
- 3.1 개요 및 파이프라인 (그림 1 참조)
- 3.2 Schema Linker (간략, 기존 기법 활용)
- 3.3 SQL Generator (간략)
- 3.4 Execution Validator (기존 축)
- 3.5 **NLI 의도 일치 검증 (핵심, RQ1)**: 역번역 → NLI entailment → ICS
- 3.6 **오류 유형별 교정 지시문 (핵심, RQ2)**: E1~E7 + E_intent, 유형별 템플릿
- 3.7 자기교정 루프 (K=3, θ=0.75)

의사코드 1개 포함 (Semantic Verifier 알고리즘).

### 4. 실험
- 4.1 데이터셋: BIRD dev + HRDB 스키마-온리 (설계 원칙 강조)
- 4.2 베이스라인: Zero-shot, DAIL-SQL, MAC-SQL (**모두 GPT-4o 재실행**)
- 4.3 지표: EX, VES, EM, **ICS(제안)** — ICS가 EX를 대체하지 않는다고 명시
- 4.4 주 결과 (표 2): 4 모델 × 2 데이터셋 × 4 지표
- 4.5 어블레이션 (표 3): A1~A4
- 4.6 오류 유형별 분석 (표 4, 그림 3)
- 통계: "paired bootstrap n=1000, 95% CI [a, b]" 형식 반복 사용

### 5. 토의
- RQ1 재확인: NLI 독립 기여 증거(A1)
- RQ2 재확인: 유형 라우팅 기여(A3)
- 한계: HRDB 스키마-온리라 데이터 분포 일반화 주장 못함, 단일 LLM(GPT-4o) 의존
- 위협 타당성: 역번역 순환성, NLI 한국어 성능, 도메인 용어 정규화 의존성

### 6. 결론
- 기여 3~4개 재진술
- 향후: 다국어 NLI, 타 LLM 백본, 실데이터 검증

## 수치 인용 규칙

1. 본문의 모든 수치는 `_workspace/03_experiment_analysis.md`에서 복사. 임의로 쓰지 않음.
2. 분석가 산출물 미완 부분은 `[DATA NEEDED: BIRD EX]` 같은 플레이스홀더 **그대로 유지**.
3. 표·그림 번호는 본문에서 "표 2", "그림 3"으로 참조. 캡션에 지표·데이터셋 명시.

## 과잉 주장 자체 검열

- "최초의" → "본 연구는 ~~를 도입한다"
- "SOTA" → "최강 베이스라인 대비 ~~ 향상"
- "완벽한" → "높은 정확도의"
- "모든 ~~에서" → "평가 대상 ~~에서"

- "혁신적" → (삭제)
