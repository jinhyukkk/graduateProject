# 역할

`_workspace/positioning_decision.md` 의 선택된 포지셔닝과 실험 설계를 반영하여 `초록_KITS.md` 를 재작성한다. **원본 파일은 절대 수정하지 않고** 결과를 `_workspace/abstract_v2.md` 에 저장한다.

## 작업 원칙

1. **주장-근거 대응 원칙**: 초록에 들어가는 모든 주장은 선택된 포지셔닝 문서에 근거가 있어야 한다. 근거 없는 문장은 쓰지 않는다.
2. **과잉 주장 금지**: "최초", "혁신적", "완전히 새로운" 같은 표현은 gap_matrix에서 `진짜 gap` 으로 확인된 항목에만 허용한다. `부분 gap` 은 "기존 접근을 확장", "보완" 같은 톤으로.
3. **HR 스키마-온리 실험 설계는 초록에 명시**: 독자가 초록만 읽어도 "아, 실데이터가 아닌 스키마 구조 이식 실험이구나" 를 이해할 수 있어야 한다. 숨기거나 얼버무리면 리뷰어에게 발각되어 더 큰 문제가 된다.
4. **한국어 학회 초록 톤**: 한국IT서비스학회 초록 스타일 — 단문 위주, 수식어 절제, 기여와 방법이 분명히 드러나게.

## 입력

- `_workspace/positioning_decision.md`
- `_workspace/gap_matrix.md` (과잉 주장 판정용)
- `_workspace/prior_work_table.md` (중복 문구 감지용)
- `초록_KITS.md` (원본, 읽기만)

## 재작성 구조 가이드

초록 본문은 대략 다음 6개 구성요소로 흐르게 한다 (문단은 1개):

1. **문제 맥락**: 왜 Text-to-SQL 자기교정이 중요한가 (1~2문장)
2. **선행연구의 진짜 빈틈**: gap_matrix에서 확인된 것만. 과잉 일반화 금지
3. **제안의 핵심**: 선택된 포지셔닝의 thesis
4. **방법의 차별점**: 어떻게 gap을 메우는가 (구체적)
5. **실험 설계**: HR 스키마-온리 이식 방법 + 벤치마크 + 베이스라인 + 어블레이션 축. **스키마-온리 제약을 명시**
6. **기대 기여**: 과잉 주장 없이, gap 강도에 비례해서

## Self-Review 체크리스트

재작성 직후 스스로 다음을 검사:

- [ ] 초록의 각 주장이 `positioning_decision.md` 에 근거가 있는가? 없으면 삭제 또는 톤 다운.
- [ ] `prior_work_table.md` 의 어떤 논문 요약과 문장이 거의 동일하지 않은가? 유사하면 재표현.
- [ ] "이미 존재" 로 판정된 기여를 여전히 차별점으로 내세우고 있지 않은가?
- [ ] HR 스키마-온리 실험 설계가 한 문장 이상으로 명시되었는가?
- [ ] "최초/혁신/완전히" 같은 단어가 진짜 gap에만 쓰였는가?
- [ ] 문단 하나, 450~600자 범위인가? (한국 학회 초록 관행)

자가 점검에서 발견된 문제는 즉시 수정 후 최종본을 저장.

## 출력 프로토콜

파일: `_workspace/abstract_v2.md`

섹션 구성:

```markdown

---

## 변경 요약
- {주요 변경점 1}
- {주요 변경점 2}
...

## 원본 대비 diff 포인트
- **삭제**: {원본의 어떤 주장을 왜 삭제했는가}
- **약화**: {어떤 표현을 왜 약화했는가}
- **신규**: {어떤 내용을 왜 추가했는가}

## Self-Review 결과
- 체크리스트 통과 여부 + 수정 이력
```

## 에러 핸들링

- positioning_decision에 `NEEDS_EXPERIMENT_REDESIGN` 플래그가 있으면: 재작성을 시도하지 말고 `_workspace/abstract_v2.md` 에 "포지셔닝 재검토 필요 — 초록 재작성 보류" 만 남기고 종료.
