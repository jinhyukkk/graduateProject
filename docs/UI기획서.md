기술 스택은 **Streamlit**을 추천해. 이유는 단순해 — 발표 15분 중 데모에 쓸 수 있는 시간이 3~5분인데, 이 짧은 시간에 보여줘야 할 건 "파이프라인 동작 과정"이지 UI 완성도가 아니야. Streamlit이면 Python 단일 파일로 LangGraph 파이프라인과 직결되고, 구현 시간도 1~2일이면 충분해. React+FastAPI는 데모 퀄리티는 높지만 공수 대비 졸업프로젝트+학술대회 일정에서 비효율적이야.

**UI 구성 요소**

전체 화면은 크게 세 영역이면 돼.

**① 입력 영역 (좌측 사이드바)**

DB 선택 드롭다운이 최상단에 와야 해. BIRD 데이터베이스 목록(california_schools, financial, thrombosis_prediction 등 주요 DB 5~6개)과 커스텀 DB(HR_Demo)를 함께 넣어. DB를 선택하면 해당 스키마 요약(테이블 수, 주요 테이블명)이 사이드바 하단에 표시되면 좋아.

자연어 질의 입력창은 메인 영역 상단에 배치. 예시 질의 버튼을 3~4개 미리 만들어두면 데모 때 타이핑 시간을 절약할 수 있어. BIRD용 예시("Find the average age of patients diagnosed with thrombosis")와 HR_Demo용 예시("부서별 평균 연봉이 가장 높은 상위 3개 부서를 조회해줘") 각각 준비해두면 돼.

**② 파이프라인 실행 과정 (메인 영역)**

이게 데모의 핵심이야. 심사자가 봐야 할 건 "각 에이전트가 순서대로 동작하고, 오류 발생 시 교정 루프가 돈다"는 과정이거든.

파이프라인 단계를 시각적으로 보여주는 방법으로, Streamlit의 `st.status`나 `st.expander`를 활용해서 각 단계가 순차적으로 펼쳐지는 구조를 만들어.

Schema RAG 단계에서는 검색된 테이블·컬럼 목록과 유사도 점수를 보여줘. 심사자가 "아, 벡터 검색으로 관련 스키마를 동적으로 가져오는구나"를 한눈에 파악할 수 있어야 해.

NL2SQL 단계에서는 생성된 SQL을 신택스 하이라이팅으로 표시. Streamlit의 `st.code(sql, language="sql")`이면 충분해.

Validator 단계에서는 SQL 실행 결과(성공/실패)와 실행 결과 테이블 미리보기를 보여줘. 정답과 비교해서 일치/불일치 여부를 초록/빨강 배지로 표시하면 직관적이야.

여기서 **정답인 경우** → "교정 불필요" 표시 후 최종 결과로 바로 이동.

**오류인 경우** → Error Classifier 단계가 활성화. 이게 데모에서 가장 중요한 순간이야. 분류된 오류 유형(예: "Condition Error")을 큰 글씨로 보여주고, 분류 근거(Classifier의 reasoning)도 expander로 펼쳐볼 수 있게 해.

적응적 교정 단계에서는 배정된 전략(CoT 또는 RAG)을 명시하고, CoT면 추론 과정을, RAG면 주입된 evidence 텍스트를 보여줘. 교정된 SQL을 원본 SQL과 diff 형태로 비교할 수 있으면 금상첨화인데, 최소한 교정 전/후 SQL을 나란히 보여주는 것만으로도 충분해.

Executor 단계에서는 교정된 SQL의 실행 결과와 정답 비교. 교정 성공이면 초록 배지 + "교정 성공", 실패면 빨강 배지.

**③ 결과 요약 (하단)**

최종 SQL, 실행 결과 테이블, 교정 이력(몇 라운드 교정했는지, 어떤 유형으로 분류됐는지)을 한 패널에 정리. 데모 끝에 한 화면으로 "이 질의는 Condition 오류 → RAG 교정 → 1라운드 만에 성공"이 요약되면 발표 흐름이 깔끔해져.

**커스텀 DB(HR_Demo) 구성 팁**

Oracle HR을 그대로 쓰면 접속 환경 문제가 생길 수 있으니, SQLite로 HR 데모 DB를 별도로 만들어두는 게 안전해. Oracle HR의 핵심 테이블(employees, departments, jobs, job_history) 4~5개를 SQLite로 옮기고, 샘플 데이터 50~100건 정도 넣으면 돼. evidence도 HR 도메인용으로 2~3개 만들어두면 좋아. 예를 들어 "연봉(salary) 컬럼은 월 기본급 기준이며 성과급은 별도 테이블(bonuses)에 저장됨" 같은 도메인 지식. 이러면 Condition/Logic 오류에서 RAG가 이 evidence를 가져와서 교정하는 걸 보여줄 수 있어.

데모 시나리오는 두 개만 준비하면 돼. BIRD에서 한 개(오류 발생 → 교정 성공 케이스), HR_Demo에서 한 개(한국어 질의 → 오류 → evidence 기반 RAG 교정). 특히 두 번째 시나리오에서 "실무 HR 데이터에서도 동작한다"를 보여주면 학회 청중에게 강한 인상을 줄 수 있어.

**우선순위 정리**

시간이 부족하면 커스텀 DB 시연은 빼도 돼. 필수는 BIRD 기반 파이프라인 동작 데모이고, HR_Demo는 보너스야. 그리고 UI 자체보다 **미리 녹화한 데모 영상**을 백업으로 준비해두는 걸 강력 추천해. 발표 현장에서 API 호출이 느리거나 네트워크 문제가 생기면 영상으로 대체할 수 있거든.
