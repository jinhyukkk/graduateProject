# SQL→NL 역번역 + NLI 검증이란?

**한 줄 요약**: LLM이 만든 SQL을 다시 자연어로 풀어 써보고, 원래 질문과 "같은 말인지" 기계로 대조하는 검사.

## 왜 필요한가

기존 Text-to-SQL 자기교정은 대부분 이렇게 검증한다:
- SQL 문법이 맞는지 (파싱 OK?)
- DB에서 실행이 되는지 (에러 안 나나?)
- 결과가 비어있지 않은지

문제: **문법도 맞고 실행도 되는데 "엉뚱한 걸 묻는" SQL**은 못 잡는다.

예시:
- 질문: "2024년에 **퇴사한** 직원 수는?"
- 생성된 SQL: `SELECT COUNT(*) FROM employees WHERE hire_date LIKE '2024%'`
- 실행 성공, 숫자도 나옴 → 기존 방식은 통과시킴
- 실제로는 **입사자**를 센 거라 완전히 틀림

## 어떻게 검증하는가 (2단계)

**1단계 — SQL→NL 역번역 (Back-translation)**
LLM에게 생성된 SQL을 보여주고 "이 쿼리가 무엇을 묻고 있는지 자연어로 설명해줘"라고 시킴.
→ "2024년에 입사한 직원의 수를 세는 쿼리입니다"

**2단계 — NLI (Natural Language Inference) 일치 검증**
원래 질문과 역번역 문장을 NLI 모델에 넣어 세 가지 중 하나로 판정:
- **entailment** (일치): 같은 뜻 → 통과
- **contradiction** (모순): 다른 뜻 → 재생성 필요
- **neutral** (애매): 경계값 → 임계값으로 결정

위 예시는 "퇴사"와 "입사"가 **contradiction**으로 잡혀서 재교정 루프로 넘어감.

## 비유

번역가가 한국어를 영어로 옮긴 뒤, **다시 한국어로 되돌려 번역해서 원문과 비교하는** 왕복 검증과 같다. 왕복했을 때 뜻이 보존되면 번역이 충실했다는 신호다.

## 이 논문에서 왜 novelty인가

선행연구 9편 중:
- CHESS: 자연어 unit test 생성 (유사하나 test case 생성 방식)
- SQLDriller: SMT 동치성 + LLM NL 실행 (데이터셋 정제용, 추론 시점 아님)
- 나머지 7편: 실행 결과·문법·rubber-duck 수준

→ **"생성 SQL을 자연어 공간으로 되돌려 원 질문과 NLI로 직접 대조"하는 구조는 부재**. 이게 gap_matrix에서 유일하게 1.0 보유 논문이 0편으로 나온 축이고, 그래서 일차 차별점으로 선택됐다.
