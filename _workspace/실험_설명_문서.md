## 1. 시스템 전체 개요

### 연구 목적

기업 인사관리 시스템(HR-DB)에서 **SQL을 모르는 일반 직원**도 자연어로 데이터를 조회할 수 있도록, LLM 기반 자기교정(Self-Correction) Text-to-SQL 파이프라인을 설계하고 구현함.

### 핵심 아이디어: 자기교정 루프

```
자연어 질의
    ↓
SQL 생성  ← LLM이 처음 생성 (오류 가능)
    ↓
실행 검증  ← 실제 DB에서 실행해서 오류 확인
    ↓
의미 검증  ← 질의 의도와 SQL이 맞는지 NLI로 확인
    ↓
오류 있으면 → SQL 교정 (최대 3번 반복)
    ↓
최종 결과 + 자연어 설명 반환
```

### 시스템 구성 3-tier

| 구성 | 기술 | 역할 |
|------|------|------|
| **Frontend** | React + TypeScript | 실험 인터페이스, 결과 시각화 |
| **Backend** | FastAPI (Python) | SC-TSQL 파이프라인 API 제공 |
| **Core** | Python 모듈 (src/) | 파이프라인 핵심 로직 |

### 2-1. 단일 쿼리 실행 화면 (Query Page / `/`)

**목적:** 개발 중 빠른 테스트 및 파이프라인 동작 확인

#### 화면 구성 요소

```
┌──────────────────────────────────────────────────────────┐
│  SC-TSQL 대시보드                              [메뉴]    │
├──────────┬───────────────────────────────────────────────┤
│          │  [1] 데이터베이스 선택기                       │
│  사이드  │  ─────────────────────────────────────────   │
│  메뉴    │  [2] 자연어 질의 입력                          │
│          │                                               │
│  - 쿼리  │  [3] 파이프라인 진행 시각화                   │
│  - 실험  │     Step 1 → Step 2 → Step 3 → Step 4       │
│  - 결과  │                                               │
│  - 비교  │  [4] 생성된 SQL + 구문 강조                   │
│          │                                               │
│          │  [5] 교정 이력 타임라인                        │
│          │                                               │
│          │  [6] 실행 결과 테이블                          │
│          │                                               │
│          │  [7] 자연어 결과 설명                          │
└──────────┴───────────────────────────────────────────────┘
```

#### 각 영역 상세 설명

**[1] 데이터베이스 선택기 (DatabaseSelector)**
- 사용 가능한 SQLite 데이터베이스 목록을 드롭다운으로 표시
- HR-DB (기업 인사관리) 또는 BIRD 벤치마크의 개별 DB 선택 가능
- 백엔드 `GET /api/databases` API에서 데이터 가져옴
- 선택한 DB가 파이프라인 전체에 적용됨

**[2] 자연어 질의 입력 (QueryInput)**
- 사용자가 자연어로 질문을 입력하는 텍스트 영역
- 예: "2024년에 입사한 영업팀 직원의 평균 연봉은?"
- 제출 버튼 클릭 시 `POST /api/query`로 전송

**[3] 파이프라인 진행 시각화 (PipelineVisualizer)**
- 파이프라인의 4개 단계를 단계별로 시각화
- 각 단계가 완료되면 상태 표시가 업데이트됨 (대기 → 진행 중 → 완료)
- 사용자가 현재 어느 단계가 실행 중인지 직관적으로 파악 가능

```
[Schema Linker] → [SQL Generator] → [Self-Correction] → [Explainer]
      ✓                 ✓              🔄 (2라운드)           ✓
```

**[4] 생성된 SQL (SqlDisplay)**
- 최종 생성(또는 교정)된 SQL을 SQL 구문 강조로 표시
- 코드블록 형식으로 복사 가능
- 교정 전 초기 SQL vs 최종 SQL을 비교하여 볼 수 있음

**[5] 교정 이력 타임라인 (CorrectionTimeline / CorrectionStepper)**
- 자기교정 과정을 단계별로 타임라인 형식으로 표시
- 각 라운드마다 다음 정보 표시:
  - 감지된 오류 유형 (예: E3_NO_SUCH_COLUMN)
  - 실행 오류 메시지
  - 의미 일관성 점수 (NLI 점수, 예: 0.68)
  - 교정된 SQL

| 라운드 | 오류 유형 | NLI 점수 | 조치 |
|--------|-----------|----------|------|
| Round 1 | E3: 컬럼 없음 | - | SQL 재생성 |
| Round 2 | 의미 불일치 | 0.68 < 0.75 | 부서 필터 추가 |
| Round 3 | - | 0.91 ✓ | 완료 |

**[6] 실행 결과 테이블 (ResultTable)**
- 최종 SQL 실행 결과를 표 형식으로 표시
- 컬럼명 + 데이터 행
- 행 수가 많을 경우 페이지네이션 적용

**[7] 자연어 결과 설명 (ResultExplanation)**
- GPT-4o가 생성한 비전문가용 자연어 설명
- SQL이나 기술 용어 없이 순수 한국어로 결과 전달
- 교정이 있었으면 "초기 쿼리에 오류가 있어 자동 수정되었습니다" 등 안내

**[8] 선택된 스키마 정보 (SchemaContextDisplay)**
- Schema Linker가 선택한 테이블/컬럼 목록 표시
- 어떤 스키마를 참조하여 SQL을 생성했는지 투명하게 공개

### 2-2. 실험 설정 화면 (Experiment Page / `/experiment`)

**목적:** 대규모 자동화 평가 실험 구성 및 실행

#### 화면 구성 요소

```
┌─────────────────────────────────────────────────────────┐
│  실험 설정                                              │
├───────────────────────────┬─────────────────────────────┤
│  [1] 실험 기본 설정 폼    │  [4] 실험 목록 패널         │
│                           │                             │
│  - 데이터셋 선택          │  실험 ID │ 상태 │ EX        │
│  - 모델 선택              │  exp_001 │ 완료 │ 69.3%    │
│  - 최대 교정 횟수 (K)     │  exp_002 │ 실행중│ -        │
│  - 의미 임계값 (θ)        │  exp_003 │ 대기 │ -        │
│  - 샘플 수                │                             │
│                           │                             │
│  [2] K-Sweep 설정 패널    │  [5] 진행률 표시            │
│                           │  ████████░░░░ 67%           │
│  K 범위: 1 ~ 5            │  현재: 101/150 질의         │
│  [K-Sweep 실행 버튼]      │                             │
│                           │  [6] 실시간 로그 스트림     │
│  [3] 실험 시작 버튼       │  [INFO] Query 102 완료...   │
│                           │  [INFO] EX: 0.693...        │
└───────────────────────────┴─────────────────────────────┘
```

#### 각 영역 상세 설명

**[1] 실험 기본 설정 폼 (ExperimentForm)**

사용자가 조정할 수 있는 하이퍼파라미터:

| 설정 항목 | 기본값 | 설명 |
|-----------|--------|------|
| 데이터셋 | hrdb | hrdb 또는 bird |
| LLM 모델 | gpt-4o-2024-11-20 | SQL 생성에 사용할 모델 |
| 최대 교정 횟수 K | 3 | 자기교정 루프 최대 반복 횟수 |
| 의미 임계값 θ | 0.75 | NLI 점수 기준값 (이 이하면 불일치 판정) |
| 샘플 수 | 150 | 평가할 질의 개수 |

**[2] K-Sweep 설정 패널 (KSweepPanel)**
- K값(최대 교정 횟수)을 범위로 지정하여 일괄 실험 수행
- 예: K=1, K=2, K=3, K=4, K=5를 모두 실험하여 최적 K 탐색
- `POST /api/experiments/k-sweep` API 호출
- 연구 논문의 Figure 2 (K에 따른 성능 변화 그래프)를 위한 데이터 수집

**[3] 실험 시작 버튼**
- `POST /api/experiments` API 호출하여 실험 생성
- 백그라운드에서 비동기로 실험 실행 (UI가 블로킹되지 않음)

**[4] 실험 목록 패널**
- 지금까지 생성된 모든 실험의 목록
- 상태: 대기(pending) / 실행 중(running) / 완료(completed) / 실패(failed)
- 클릭하면 해당 실험의 결과 페이지로 이동

**[5] 진행률 표시 (ProgressBar)**
- 현재 실험의 진행 상황을 실시간으로 표시
- WebSocket(`/ws/experiments/{id}/progress`)으로 서버에서 푸시
- 전체 질의 수 대비 완료된 질의 수 표시

**[6] 실시간 로그 스트림 (LogStream)**
- 실험이 진행되면서 발생하는 로그를 실시간으로 출력
- 각 질의별 처리 결과, 오류 발생 현황, 현재 누적 메트릭 표시
- WebSocket 연결로 서버에서 실시간 스트리밍

### 2-3. 실험 결과 화면 (Results Page / `/results/:id`)

**목적:** 완료된 실험의 전체 성과를 시각화하고 분석

#### 화면 구성 요소

```
┌──────────────────────────────────────────────────────────┐
│  실험 결과: exp_001 (HR-DB, K=3, θ=0.75)               │
├────────────────┬───────────────────────┬─────────────────┤
│  [1] 메트릭    │  [2] 메트릭 카드       │                 │
│      카드      │                       │                 │
│                │  EX: 69.3%           │                 │
│  EX  CSR  시간 │  CSR: 77.5%          │                 │
│               │  Avg Latency: 4.2s    │                 │
├────────────────┴───────────────────────┴─────────────────┤
│  [3] 교정 라운드별 실행 정확도 추이 (꺾은선 그래프)      │
│                                                          │
│  100% ─────────────────────────────────────────         │
│   80%          ┌───────────────                         │
│   60%  ────────┘                                        │
│   40%                                                    │
│        초기   R1    R2    R3 (최종)                    │
├──────────────────────────────────────────────────────────┤
│  [4] 오류 유형별 발생 빈도 (막대 그래프)                 │
│                                                          │
│  E1 ██                                                   │
│  E2 ████████                                             │
│  E3 ██████                                               │
│  E4 ████                                                 │
│  E5 ██                                                   │
├──────────────────────────────────────────────────────────┤
│  [5] 쿼리별 상세 결과 테이블                             │
│                                                          │
│  # │ 질의 │ 정답? │ 교정횟수 │ 오류유형 │ 소요시간 │ 보기│
│  1 │ 입사한 직원... │ ✓ │ 0 │ - │ 2.1s │ →  │
│  2 │ 부서별 연봉... │ ✓ │ 2 │ E3 │ 8.4s │ →  │
│  3 │ 퇴직률이... │ ✗ │ 3 │ E7 │ 12.1s│ →  │
└──────────────────────────────────────────────────────────┘
```

#### 각 영역 상세 설명

**[1][2] 메트릭 카드 (MetricCard)**

3가지 핵심 평가 지표를 한눈에 표시:

| 지표 | 의미 | 계산 방식 |
|------|------|-----------|
| **EX (Execution Accuracy)** | 실행 정확도 | 예측 SQL 실행 결과 = 정답 SQL 결과인 쌍 수 / 전체 |
| **CSR (Correction Success Rate)** | 교정 성공률 | 오류가 났지만 교정에 성공한 질의 수 / 초기 오류 질의 수 |
| **Avg Latency** | 평균 응답 시간 | 전체 질의의 평균 소요 시간(초) |

**[3] 교정 라운드별 EX 추이 (CorrectionLineChart)**
- X축: 초기 생성 → Round 1 교정 → Round 2 교정 → Round 3 교정
- Y축: 누적 실행 정확도(%)
- 교정이 반복될수록 정확도가 어떻게 향상되는지 시각화
- **논문 핵심 그래프**: 자기교정의 효과를 직접 보여주는 증거

**[4] 오류 유형별 발생 빈도 (ErrorTypeBarChart)**
- 8가지 오류 유형(E1~E8)별로 발생 빈도를 막대 그래프로 표시
- 어떤 오류가 가장 많이 발생하는지 파악 가능
- 모델이나 설정 변경 시 오류 분포 변화를 비교하는 데 활용

**오류 유형 정의:**

| 유형 | 이름 | 분류 | 예시 |
|------|------|------|------|
| E1 | SYNTAX | 구문 | SELECT ,, FROM ... |
| E2 | NO_SUCH_TABLE | 의미 | FROM 없는_테이블 |
| E3 | NO_SUCH_COLUMN | 의미 | SELECT 없는_컬럼 |
| E4 | AMBIGUOUS_COLUMN | 의미 | 두 테이블에 같은 컬럼명 |
| E5 | TYPE_MISMATCH | 의미 | 숫자 컬럼에 문자 비교 |
| E6 | AGGREGATE_ERROR | 의미 | GROUP BY 누락 |
| E7 | EMPTY_RESULT | 논리 | 결과가 0행 |
| E8 | EXCESSIVE_RESULT | 논리 | 결과가 1000행 초과 |

**[5] 쿼리별 상세 결과 테이블 (QueryResultTable)**
- 실험에 포함된 모든 질의의 처리 결과를 테이블로 나열
- 각 행에서 볼 수 있는 정보:
  - 정답 여부 (EX 기준)
  - 교정 횟수 (0~3)
  - 발생한 오류 유형
  - 처리 소요 시간
- 행 클릭 시 쿼리 상세 분석 화면으로 이동

### 2-4. 쿼리 상세 분석 화면 (Query Detail Page / `/results/:id/query/:qid`)

**목적:** 특정 질의 하나의 파이프라인 처리 과정을 단계별로 상세 분석

#### 화면 구성 요소

```
┌──────────────────────────────────────────────────────────┐
│  질의 상세: "2024년 영업팀 입사자의 평균 연봉은?"        │
├──────────────────────────────────────────────────────────┤
│  [1] 파이프라인 단계 요약                                 │
│                                                          │
│  Schema Linker → SQL Generator → Self-Correction(2회)   │
│                                              → Explainer │
├──────────────────────────────────────────────────────────┤
│  [2] 선택된 스키마                                        │
│  테이블: employees, departments, salary_info            │
│  컬럼: emp_id, name, dept_id, join_date, salary, ...    │
├──────────────────────────────────────────────────────────┤
│  [3] 초기 생성 SQL (Round 0)                             │
│  SELECT AVG(salary) FROM employees                       │
│  WHERE join_date >= '2024-01-01'                        │
├──────────────────────────────────────────────────────────┤
│  [4] 교정 과정 (Round 1)                                 │
│  오류: E3_NO_SUCH_COLUMN (dept_name)                    │
│  역번역: "직원의 평균 급여"                              │
│  NLI 점수: 0.62 (임계값 0.75 미만 → 불일치)            │
│  진단: "부서 필터 및 JOIN 누락"                          │
│  → 교정 SQL: ... JOIN departments ON ... WHERE dept='영업'│
├──────────────────────────────────────────────────────────┤
│  [5] 최종 SQL 및 실행 결과                                │
│  SELECT AVG(s.salary) FROM employees e                   │
│  JOIN departments d ON e.dept_id = d.dept_id            │
│  JOIN salary_info s ON e.emp_id = s.emp_id             │
│  WHERE d.dept_name = '영업팀'                           │
│  AND e.join_date BETWEEN '2024-01-01' AND '2024-12-31' │
│                                                          │
│  실행 결과: avg_salary = 54,200,000                     │
├──────────────────────────────────────────────────────────┤
│  [6] 자연어 설명                                          │
```

#### 각 영역 상세 설명

**[2] 선택된 스키마 (SchemaContextDisplay)**
- Schema Linker가 42개 테이블 중 선택한 관련 테이블과 컬럼만 표시
- 외래키(FK) 관계도 포함하여 표시
- LLM이 어떤 컨텍스트를 보고 SQL을 생성했는지 투명하게 공개

**[4] 교정 과정 상세 (CorrectionStepper)**
- 각 라운드별 자세한 교정 내용:
  - **실행 오류**: SQLite에서 실행 시 발생한 실제 오류 메시지
  - **오류 분류**: E1~E8 중 어떤 유형으로 분류되었는지
  - **역번역**: SQL을 자연어로 역번역한 결과
  - **NLI 점수**: 원래 질의와 역번역의 의미 일관성 점수 (0~1)
  - **불일치 진단**: NLI 점수가 낮을 때 GPT-4o가 생성한 원인 분석
  - **교정 SQL**: 오류 정보를 반영하여 교정된 SQL

### 모듈별 핵심 기술 요약

#### Schema Linker (`src/schema_linker.py`)

**역할:** 42개 테이블 중 질의와 관련된 테이블/컬럼만 추출

**2단계 방식:**
1. **임베딩 검색**: OpenAI `text-embedding-3-large`로 질의와 스키마를 벡터화, FAISS로 코사인 유사도 상위 K개 후보 추출
2. **LLM 정제**: GPT-4o가 후보 스키마를 검토하여 최소한으로 필요한 스키마만 선택

**왜 중요한가:** LLM에게 42개 테이블 전체를 보여주면 컨텍스트 낭비 + 혼란 발생. 관련 스키마만 전달하면 정확도 향상.

#### SQL Generator (`src/sql_generator.py`)

**역할:** 선택된 스키마와 자연어 질의로 SQL 생성

**Few-shot 선택 전략:**
- 질의 의미 유사도 50% + 스키마 구조 유사도 50% 가중합
- 가장 유사한 예시 쿼리 3개를 프롬프트에 포함
- 근거: DAIL-SQL 논문의 접근법을 채택

#### Execution Validator (`src/execution_validator.py`)

**역할:** 생성된 SQL을 실제 SQLite에서 실행하고 오류를 분류

**보안 기능:**
- 읽기 전용(read-only) 모드로 실행
- 5초 타임아웃 설정
- PRAGMA query_only 활성화 → INSERT/UPDATE/DELETE 차단

**오류 분류 기준:**
- 구문 오류(E1): SQL 문법 자체 오류
- 의미 오류(E2~E6): 스키마와 불일치
- 논리 오류(E7~E8): 실행은 되지만 결과가 이상함

#### Semantic Verifier (`src/semantic_verifier.py`)

**역할:** SQL이 질의 의도를 올바르게 반영하는지 NLI로 검증

**3단계 과정:**
1. GPT-4o로 SQL을 자연어로 역번역: "이 SQL이 답하는 질문은?"
2. `cross-encoder/nli-deberta-v3-base`로 원래 질의 vs. 역번역 간 entailment 점수 계산
3. 점수 < 0.75이면 GPT-4o로 불일치 원인 진단

**왜 필요한가:** SQL이 문법적으로는 맞아도 의도와 다른 결과를 반환할 수 있음. 예: 부서 필터 없이 전체 직원 평균을 계산.

#### Corrector (`src/corrector.py`)

**역할:** 수집된 오류 정보를 바탕으로 SQL 교정

**프롬프트에 포함되는 정보:**
- 원래 자연어 질의
- 현재 오류가 있는 SQL
- 실행 오류 메시지 + 분류 (E1~E8)
- 역번역 결과 + NLI 점수
- 불일치 진단 내용

**교정의 강점:** 단순히 "오류가 있다"고 전달하는 것이 아니라, 오류 유형별 맥락과 의미 불일치 진단을 함께 제공하여 더 정확한 교정 가능.

#### Result Explainer (`src/result_explainer.py`)

**역할:** SQL 결과를 비전문가가 이해할 수 있는 자연어로 설명

**설명에 포함되는 내용:**
- 질의에 대한 직접적인 답변
- 교정이 있었으면 간단한 안내 ("초기 쿼리에 오류가 있어 수정되었습니다")
- 기술 용어 없이 순수 한국어
